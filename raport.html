1. Introduction

The main purpose of the project is to develop an application where people can contribute to making the city greener and the air quality better by indicating which areas need help.

2.2 Product Perspective

Green City is a modular web application which brings together autorities,non-profit organizations and normal citizens to get a better life quality in the cities.

2.3 Product Functions

This application provides for the user the posibility to create an account so he/she can point the areas where they want to see trees and other bushes. The user can check wheather the area he/she pointed to has been solved or not.

2.4 User Classes and Characteristics

Regular users

Can create an account,add a new zone,check the zone's progress,join a tree planting event.

Authorities

Can check new zones,updates the progress.

Non-profit Organizations

Can create an event for tree planting for a zone that the authorities approve(not all of them can be safe).

This Web aplication is compatible with any browser, it's interface provides support for big screens such as PC and mobile screens based on their dimensions, providing the users with a nice experience on their PC and on their mobile phone.

3. Ideas/concepts regarding the project design

3.1 Icons

Using relevant icons is a big part of designing an application,for this reason i choose simple icons that leave no room for interpretation.

  • For the name of the application i used an icon with trees which gives the user a sneak peek about what type of app this is.
  • For the phone version of the app,the responsive part, i used a hamburger menu icon to hide the menu buttons. This is a common practice that makes the design easy and user friendly.
  • On the home page we have an interactive map which has a map marker icon to indicate that someone put a "mark" on a certain area. The icon is pretty well-known,we can see it all the time in different GPS apps.

Also it is important that all the icons have the same style,for a more put together look, for this app all the icons i used are outlined.

3.2 Buttons

Important aspects that i used in the prototype buttons are:

  • Making buttons look like buttons. Buttons need to look buttonyâ€”with a bit of shading on their surface or around their edges to make them stand out from the background and look clickable.A shape around a word without any shading isn't enough to make it clearly look buttony, and a simple word on its own definitely doesn't provide enough visual clues.
  • Make the most important button look like it's the most important one. Another common way of making it difficult for users to play hunt-for-the-button-I-want is to offer them lots of buttons in a neat row, all of them looking very similar. For example on the "Add a new area" page we have a form with more buttons,one for uploading pictures and the main button which adds the area to the database. The design is very intuitive and makes it very easy for the user to know which button is the most important because the "Add" button is significantly bigger than the other one.
  • Label buttons with what they do. It is important that buttons have suggestive labels. For example if a button leads to a sign-up or a log-in page we can't name it "Log-out".

3.3 Layout

A page is normally read from left to right by the user. As a result, on the left side of the page that displays all of the solved locations, there is more information. This strategy has a direct visual influence on the user, since you can see some pictures first and then the information. The content will be concise, containing only the most necessary information, making it easier to read for the user.

4. Examples concerning the QOC (Questions, Options, Criteria) method

4.1 Sign up form

4.1.1 Page description

The page where the user creates an account on the app.

4.1.2 QOC method and thought process

How can I get the user to sign up for an account? I have the option of creating my own authentication or using a third-party authentication service such as Google or Github. I chose to do my own authentication for this procedure and not let the user pick between the two options so that I could handle the database and schema easily.

4.2 Adding a new area

4.2.1 Page description

The page for adding a new area let's the user put all the information needed to pin the point on the map,in other words they will fill in the address.

4.2.2 QOC method and thought process

How can I get the user to type in the address? I have a few options: making a list, utilizing Google Maps, and manually typing in the address. Because of the nature of my program, I decided to employ two options, allowing the user to select whether to manually put in the address or use the current location via Google Maps. I chose these two alternatives because users can add any area that needs to be refreshed, which means it might be anywhere near their home where they know the address or somewhere else where they don't.

4.3 Solved location page

4.3.1 Page descriptio

The page that displays all the solved areas.

4.2.2 QOC method and thought process

How can I create a page that displays everything while remaining interesting? I can use different colors for each displayed place, I can use images to bring attention to the location, and I can only show information about the location. I choose to combine the first and last options for easy user readability and to capture attention. I show the photos first, followed by some quick facts about the site, including the address.
